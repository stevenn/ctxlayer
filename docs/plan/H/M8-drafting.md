# M8 — Catalogue diff + AI-assisted drafting

> Implementation spec for [[H]] M8. Depends on M7a worker (skills
> tables) + M7c CLI (`@ctxlayer/cli` shipped). Drafts run on the
> operator's Claude Code CLI, not on the worker.
> Status: design, not yet implemented.

## Scope

- **Catalogue diff pipeline**: detect `inputSchema` changes per upstream
  tool, persist a "last schema change" timestamp.
- **Skill staleness flag** (computed at read time): a skill is stale if
  any attached upstream tool's schema changed after the skill's
  `updated_at`.
- **`/api/skills/draft-context` endpoint**: assembles a JSON context
  bundle the CLI inlines into the drafter prompt. No LLM on the
  worker.
- **`ctxlayer draft-skill` CLI command**: fetches the bundle, shells
  `claude -p`, prompts the operator to save the draft as a `status='draft'`
  skill.
- **Schema-reference linter** (server-side, on `POST /api/skills`):
  flag tool / argument references in the body that don't exist in the
  current `tools/list`.
- **Admin SPA**: stale badges, copy-command helper enriched to
  reflect upstream/tool context, draft-review affordances.

Out of scope: Tier 6 proactive suggestions (M10+), agentic drafting
via `--mcp-config` (deferred power-user variant).

## Migration `0012_skill_drafting.sql`

Two additive columns, no new tables:

```sql
-- Track when an upstream tool's inputSchema actually changed.
-- NULL = never changed since first cache or schema-diff tracking
-- predates the row. Updated by the refresh path only when the new
-- hash differs from the cached hash.
ALTER TABLE upstream_tools ADD COLUMN last_schema_change_at INTEGER;

-- Optional drafter provenance. NULL for manually-authored skills;
-- populated by `POST /api/skills` when the request carries a
-- `drafter_meta` field (the CLI draft-skill path does).
-- Stored as JSON text: { from: 'cli+claude-code', model: '...',
--   modelVersion: '...', contextInputs: ['rag','usage'], at: <epoch> }.
ALTER TABLE skills ADD COLUMN drafter_meta TEXT;
```

No `CHECK` on `drafter_meta` (free JSON); the linter validates shape
via Zod on write.

## Catalogue diff pipeline

### Where it hooks

Existing refresh path: `apps/worker/src/upstream/catalogue.ts:60-82`
calls `replaceCachedTools()` at `db/queries/upstreams.ts:332-351`.
Replace the unconditional `DELETE old + INSERT new` with a
**3-phase update inside one `env.DB.batch()`**:

1. SELECT current `(tool_name, input_schema_hash)` rows for the
   upstream.
2. Compute `newHash = sha256(canonicalised(inputSchema))` per
   incoming tool. (Canonicalise = JSON.stringify with sorted keys to
   stabilise.)
3. For each tool:
   - If `newHash !== oldHash` → INSERT/REPLACE row + set
     `last_schema_change_at = <now>`.
   - If `newHash === oldHash` → INSERT/REPLACE row + leave
     `last_schema_change_at` untouched. (Still bumps `cached_at`.)
4. DELETE rows for tools no longer in the new list.

`input_schema_hash` could be a computed column or a stored TEXT
column populated alongside `input_schema`. Cleanest: store
`input_schema_hash TEXT` for fast comparison without re-canonicalising
on every read.

**Schema addendum to `0012_skill_drafting.sql`:**

```sql
ALTER TABLE upstream_tools ADD COLUMN input_schema_hash TEXT;
```

Defaults NULL on existing rows; populated on next refresh. Until then,
those rows are treated as "never changed" (NULL `last_schema_change_at`).

### Diff event log (optional)

H mentions a "diff summary" for the admin SPA. Two ways:

- **(A) Inline column** `last_diff_summary TEXT` — overwrites; only
  the latest diff is visible.
- **(B) Separate table** `upstream_tool_diff_events (upstream_id,
  tool_name, diffed_at, change_summary)` — full history, queryable.

**Recommend A for M8** — admin only needs "what changed last" to
decide whether to refresh a skill. Full history is a v2 nice-to-have;
defer.

Addendum:

```sql
ALTER TABLE upstream_tools ADD COLUMN last_diff_summary TEXT;
```

Populated alongside `last_schema_change_at`. Summary format: small
human string like `"added required: parent_id; removed: priority"`,
generated by a server-side diff helper in `upstream/schema-diff.ts`.

## Skill staleness — computed at read time

No new column on `skills`. `listSkills` and `getSkillById` LEFT JOIN
attachments + `upstream_tools` and compute:

```sql
SELECT s.*,
  EXISTS (
    SELECT 1 FROM skill_attachments sa
    JOIN upstream_tools ut
      ON ut.upstream_id = sa.upstream_id
     AND ut.tool_name   = sa.tool_name
    WHERE sa.skill_id = s.id
      AND sa.tool_name != ''                              -- whole-upstream attachments don't gate staleness
      AND ut.last_schema_change_at IS NOT NULL
      AND ut.last_schema_change_at > s.updated_at
  ) AS is_stale
FROM skills s
WHERE s.deleted_at IS NULL;
```

`is_stale` returned in `SkillSummary` and `SkillDetail` DTOs (add as
optional field; defaults to `false`). SPA shows a yellow badge on
stale rows. MCP `list_skills` does **not** filter or annotate stale —
agents shouldn't second-guess the operator.

## `/api/skills/draft-context` endpoint

### Shape

`GET /api/skills/draft-context?upstream=<slug>&tool=<name>&prompt=<text>`

| Param | Required | Purpose |
|---|---|---|
| `upstream` | yes | upstream slug (DB lookup) |
| `tool` | no | tool name; if absent, bundle covers the whole upstream |
| `prompt` | no | freeform operator prompt; echoed back into the bundle so the CLI prompt template can use it |

Auth: `requireAdmin` (drafting is admin-gated; non-admins can't
publish anyway). No `requireCsrf` — GET is safe.

Response (`DraftContextBundle` Zod schema):

```ts
{
  upstream: {
    slug: string;
    displayName: string;
    transport: 'http' | 'sse';
  };
  focusTool: {
    name: string;
    description: string | null;
    inputSchema: unknown;          // JSON
    lastSchemaChangeAt: number | null;
  } | null;                        // null if no --tool param
  allTools: Array<{                // light catalogue for cross-references
    name: string;
    description: string | null;
  }>;
  relatedDocs: Array<{             // top-k RAG excerpts from doc corpus
    slug: string;
    title: string;
    excerpt: string;               // ~500 chars from the relevant chunk(s)
    relevanceScore: number;
  }>;
  usageAggregates: {               // {user, upstream, tool} usage stats
    totalCalls: number;            // last 30d
    callsByDay: Array<{day: string; count: number}>;
    topArgPatterns: Array<{        // most common arg shapes
      argSummary: string;          // e.g. "team_id=ENG, labels=triage"
      count: number;
    }>;
  } | null;                        // null if no usage events
  styleSkills: Array<{             // 2-3 existing published skills as style refs
    slug: string;
    title: string;
    body_md: string;
  }>;
  operatorPrompt: string | null;   // echo of ?prompt= param
  generatedAt: number;             // epoch
}
```

### Server-side assembly

New file: `apps/worker/src/api/skills-draft-context.ts`

```ts
async function buildBundle(env, { upstreamSlug, toolName, operatorPrompt, userId }) {
  const upstream = await getUpstreamBySlug(env, upstreamSlug);
  if (!upstream) throw new HttpError(404, 'upstream_not_found');

  const tools = await listUpstreamTools(env, upstream.id);    // existing query
  const focusTool = toolName ? tools.find(t => t.name === toolName) ?? null : null;

  // RAG: query Vectorize for top-3 doc chunks mentioning the upstream slug
  //   + (if focusTool) the tool name. Reuses existing search infra.
  const relatedDocs = await searchDocsForBundle(env, {
    query: `${upstream.slug} ${focusTool?.name ?? ''}`.trim(),
    k: 3,
  });

  // Usage aggregates: last 30d for (user, upstream, tool); reuses
  // usage_rollups_daily where possible, falls back to usage_events for argShapes.
  const usageAggregates = await buildUsageAggregates(env, {
    userId, upstreamId: upstream.id, toolName,
  });

  // Style refs: 2 published skills already attached to this upstream
  // (or, if zero, 2 most-recently published of any upstream).
  const styleSkills = await pickStyleSkills(env, upstream.id, 2);

  return DraftContextBundle.parse({
    upstream: { slug: upstream.slug, displayName: upstream.displayName, transport: upstream.transport },
    focusTool: focusTool ? {
      name: focusTool.name, description: focusTool.description,
      inputSchema: focusTool.inputSchema, lastSchemaChangeAt: focusTool.lastSchemaChangeAt,
    } : null,
    allTools: tools.map(t => ({ name: t.name, description: t.description })),
    relatedDocs, usageAggregates, styleSkills,
    operatorPrompt, generatedAt: Math.floor(Date.now() / 1000),
  });
}
```

Bundle size: tooled to stay under ~32 KB so it inlines cleanly into a
`claude -p` prompt without straining context. If a real bundle blows
past that — truncate the longest field (usually `relatedDocs[*].excerpt`).

## `ctxlayer draft-skill` CLI command

### Signature

```
ctxlayer draft-skill <upstream> [--tool <name>] [--prompt <text>]
                     [--from-suggestion <id>]  (M10+ Tier 6, not in M8)
                     [--no-save]               (preview only, never POST)
                     [--system-prompt-file <path>]   (override default drafter prompt)
```

### Flow

1. Locate `claude` on PATH. If missing:
   ```
   error: ctxlayer draft-skill needs Claude Code installed.
     Install: https://claude.com/claude-code
     Or author manually at https://<your-base-url>/app/admin/skills
   ```
   Exit 2.
2. Load credentials, refresh if expired.
3. `GET /api/skills/draft-context?upstream=...&tool=...&prompt=...`
   → bundle.
4. Assemble drafter system prompt (lives at
   `packages/cli/src/drafter/system-prompt.ts` — a multi-line string
   describing skill structure, the output JSON schema, and how to use
   the bundle's input sections).
5. Assemble the user prompt as stdin:
   ```
   <bundle-as-pretty-json>
   ---
   Operator request: ${operatorPrompt ?? '(none — propose a useful skill from the context above)'}
   ```
6. Shell out:
   ```
   claude -p \
     --bare \
     --no-session-persistence \
     --output-format json \
     --json-schema "$(cat dist/schemas/draft-output.json)" \
     --system-prompt "$systemPrompt" \
     --max-budget-usd 0.50 \
     --tools "" \
     <<<"$userPromptStdin"
   ```
   Use `spawn` (not `exec`) for stream-friendly stdio. Capture stdout +
   stderr.
7. Parse Claude's JSON response. Schema:
   ```json
   {
     "frontmatter": {
       "title":       "string (40-80 chars)",
       "slug":        "string (lowercase-hyphen)",
       "description": "string (single-line, when-to-use)",
       "triggerText": "string (optional)"
     },
     "body": "string (markdown)"
   }
   ```
8. Render a terminal preview:
   ```
   ──── Draft preview ────
   Slug:        linear-triage
   Title:       Linear customer-bug triage
   Description: When a customer reports a bug, file it in Linear ENG team with triage labels.
   ──── Body ────
   # Triaging a customer bug ...
   ──── ─────── ────
   ```
9. Prompt operator:
   ```
   Save as draft? (s)ave / (e)dit / (p)review again / (a)bort:
   ```
   - **Save**: `POST /api/skills` with `{...frontmatter, body, status:'draft', drafterMeta: {...}}`. Capture model + version from Claude's response envelope.
   - **Edit**: open `$EDITOR` (default `vi`) on a temp file containing the frontmatter + body; on save, re-prompt.
   - **Preview again**: redisplay.
   - **Abort**: exit 0 without saving.
10. On save: print URL to admin SPA edit page so operator can polish
    further before publishing.

### `drafterMeta` shape posted to the server

```json
{
  "from": "cli+claude-code",
  "model": "claude-opus-4-7",
  "modelVersion": "claude-opus-4-7-1m-202604",
  "contextInputs": ["rag", "usage", "style"],  // which bundle sections were non-empty
  "operatorPromptProvided": true,
  "claudeCodeVersion": "1.0.x",
  "draftedAt": 1717890000
}
```

The model + version come from Claude Code's JSON envelope (which
reports `model_name` in its metadata fields). If absent, fall back to
`"unknown"`.

### `POST /api/skills` extension

Adds optional `drafterMeta` field. The handler:
- Validates `DrafterMeta` via Zod (existing if posted).
- Stores serialised JSON in `skills.drafter_meta`.
- Otherwise unchanged from M7a.

## Schema-reference linter

### Where it runs

Server-side, in `POST /api/skills` and `PATCH /api/skills/:id` and
`PUT /api/skills/:id/content`. Centralised in
`apps/worker/src/skills/schema-linter.ts`:

```ts
export type LintFinding = {
  kind: 'unknown_tool' | 'unknown_arg';
  reference: string;          // the string we saw in the body
  upstreamSlug: string | null;
  toolName: string | null;
};

export async function lintSkillBody(env, body_md, attachments): Promise<LintFinding[]> {
  // Parse the body for `inline.code` patterns matching:
  //   - `<upstream>__<tool>`        (mangled MCP form)
  //   - `<tool>(arg=value, ...)`    (callable shorthand)
  // For each match, look up the attached upstream's cached schema
  // and verify the tool + arg names exist.
}
```

### Behaviour

- Findings returned alongside the saved skill: `{ skill, lintFindings }`.
- Save succeeds even with findings — they're warnings, not blockers.
  Operator might be intentionally referring to a not-yet-existing tool.
- SPA renders findings as a yellow strip above the editor.

### Why not a hard block

Skills are markdown for the agent. Agents are robust to mild reference
errors. A linter is a quality nudge, not a contract. Hard-blocking
would frustrate operators when the linter has a false positive.

## Admin SPA additions

### Stale badge

`/app/admin/skills` table row gets a yellow `Stale` badge when
`skill.isStale === true`. Hovering shows: "Upstream X tool Y schema
changed at <date>; review skill body."

`/app/admin/upstreams` expanded row (existing): tool sub-rows show
`Last schema change: <date>` next to `Last refresh:` if
`lastSchemaChangeAt != null`. Stale attached-skills chips get the
same yellow treatment.

Already plumbed in M7b. In M8, the per-tool row and per-upstream
section show a `<CopyCommandButton>` that builds:

```

```

…copying the exact command to clipboard. Helps non-CLI-fluent admins.

### Draft-review tweaks

when present:
```
Drafted by Claude Code (Opus 4.7, 2026-05-30) using rag + usage context.
```

Provenance is a quality cue for the reviewer.

## Verification

1. **Diff detection**: register a real upstream (Linear MCP). Trigger
   manual refresh via admin UI → cache rows have `cached_at` +
   `last_schema_change_at` (NULL on first ever, populated on second
   if schema actually changes). Mock a schema change by toggling
   `inputSchema` of one cached tool in D1 directly → trigger refresh
   → row's `last_schema_change_at` updates, `last_diff_summary`
   populated.
2. **Staleness flag**: create a skill attached to the mock-changed
   tool → `GET /api/skills` → `isStale: true`. Skill body un-edited
   for some time; once you save a new revision, `updated_at` > the
   change timestamp → `isStale: false`.
3. **Draft-context bundle**: `curl -H 'cookie: <admin>'
   '/api/skills/draft-context?upstream=linear&tool=create_issue&prompt=triage flow'`
   → JSON bundle includes all four sections (focusTool, allTools,
   relatedDocs, usageAggregates if any).
4. **Drafting E2E**:
   - With `claude` installed: `ctxlayer draft-skill linear --tool create_issue --prompt "triage flow"`.
   - Browser doesn't open; CLI fetches bundle, shells claude, prints
     preview.
   - Choose `s` → skill appears in `/app/admin/skills` with
     `drafterMeta` populated and `status='draft'`.
   - Without `claude`: same command → exits 2 with install pointer.
5. **Schema linter**:
   - Save a skill body that references `linear__bogus_tool` →
     response carries `lintFindings: [{kind: 'unknown_tool', ...}]`;
     SPA shows yellow strip.
   - Save with valid refs → empty findings.
6. **Copy-command helper**: SPA upstream detail page → click "Draft"
   button on a tool row → clipboard has
   `ctxlayer draft-skill linear --tool create_issue`. Paste into
   shell → command runs.
7. **`/smoke`** passes; new endpoint hits return 200.
8. **Security invariants**: drafter system prompt doesn't leak any
   sensitive info in the bundle (no IdP tokens, no upstream
   credentials). Verify by reading the bundle assembler code.
9. **Cost ceiling**: `--max-budget-usd 0.50` enforced by `claude` →
   confirm a deliberately-too-long bundle doesn't burn more than $0.50.
   (Inspect Claude Code stderr after a draft.)

## Sequencing within M8

1. **Migration + diff pipeline** — `0012`, schema-diff helper, refresh
   path update. Verifiable in isolation via mock D1 updates.
2. **Staleness in listSkills/getSkillById + SPA badge** — joins +
   DTO extension + table render. Independent of drafting.
3. **`/api/skills/draft-context` endpoint** — bundler + handler + DTO.
   Curl-testable.
4. **`ctxlayer draft-skill` command** — depends on (3); contains the
   bulk of the new code. Sub-sequence: claude-runner + system prompt
   first (mock the bundle), then wire the API client. End-to-end
   draft-then-save loop completes here.
5. **Schema linter + SPA strip** — additive; depends on M7b skill
   editor existing.
6. **Drafter-meta provenance + SPA copy** — last polish; depends on
   M7b SkillDrawer.

Each step is independently committable; SPA + worker buildable at
every checkpoint.

## Risks called out

- **Bundle size bloat**: usage aggregates + RAG excerpts can balloon.
  Add an upper bound (`maxBundleBytes = 32_000`) in
  `draft-context-bundler.ts`; truncate `relatedDocs[*].excerpt` first,
  then `usageAggregates.callsByDay` (collapse to weekly bins).
- **Claude Code JSON envelope**: confirmed `--json-schema` works
  in the `claude --help` capture. But the envelope key names
  (`result` vs `content` vs `text`) vary by `--output-format`. Spike
  on day 1 of M8 CLI work; pin to the exact path the current Claude
  Code version produces.
- **Schema linter false positives**: regexp-based parsing of markdown
  code spans will catch non-tool things like `process_id(42)`.
  Mitigation: only flag matches whose parent is an attached upstream
  slug (i.e. only check `<slug>__<tool>` patterns where `<slug>`
  matches `skill_attachments` for that skill).
- **`--max-budget-usd` is operator's API budget**: cost is borne by
  the operator's Claude subscription, not ctxlayer. Make the budget
  configurable via env var (`CTXLAYER_DRAFT_MAX_USD`) for users on
  Pro-only plans where API metering doesn't apply.
- **Drafter prompt drift**: as the team iterates on the system
  prompt, ensure version-pinning — the CLI's prompt version is part
  of `drafterMeta`. If a draft was generated under prompt v3 and the
  current prompt is v5, the SPA can show "drafted under older prompt"
  for context.
- **Diff summary quality**: simple JSON diff produces noisy summaries
  for big schemas. Acceptable for v1 — operators read the actual
  schema for the real story; the summary is a hint.

## What stays beyond M8

- **Tier 6 proactive suggestions** (M10ish): nightly cluster
  usage-event sequences via Workers AI embeddings, surface
  "suggested skill" rows in an admin inbox. Each suggestion gets a
  prebuilt `ctxlayer draft-skill --from-suggestion <id>` invocation.
- **Agentic drafting (Design B)** via `--mcp-config`: deferred per H.
  Worth revisiting only if the server-bundled context (Design C) feels
  too rigid for a real case.
- **Diff event history table**: if "what changed when, three months
  ago" becomes a real question, promote `last_diff_summary` into a
  proper events table.
